# Overleaf Manuscript Editing Workflow

When the user asks to edit, expand, reorganize, polish, or maintain the manuscript in the Overleaf project for SCIGEN+, follow this workflow.

## Primary editing targets

You may directly edit only these files unless the user explicitly expands scope:

- `/pscratch/sd/r/ryotaro/data/generative/overleaf/scigenp_overview/main.tex`
- `/pscratch/sd/r/ryotaro/data/generative/overleaf/scigenp_overview/supp_scigenp.tex`

These are the only manuscript files that should be modified by default.

## Core manuscript policy

- Treat `main.tex` as the master outline and rough memo for the intended manuscript.
- The content inside `main.tex` may be unordered, fragmentary, repetitive, or loosely written.
- Infer the intended manuscript organization from those notes and rewrite them into a coherent scientific narrative.
- You may reorganize sections, merge fragments, rewrite heavily, and improve transitions, but preserve the underlying intended content unless there is a strong technical or narrative reason to change it.
- Keep the writing publication-oriented, concise, and scientifically grounded.
- Maintain consistency between the main text and supplement.
- Preserve LaTeX compilability.
- Do not insert local filesystem paths, internal repository paths, or raw local links into the manuscript text.

## Writing goals

When editing the manuscript:

- convert rough memo text into polished scientific prose
- infer and strengthen the paper narrative
- clarify how SCIGEN+ relates to prior SCIGEN and DiffCSP++
- clearly explain what is already implemented versus what is planned
- explain the role of human preference data, AB testing, and preference-based learning carefully
- explain DPO-style fine-tuning only to the extent supported by code, notes, or clearly marked future-work framing
- remove repetition, vague phrasing, and internal memo-like wording
- make the manuscript readable as a coherent ML + materials science paper
- keep terminology consistent across the paper and supplement

## Factuality constraints

- Do not hallucinate experiments, numerical results, implementation status, datasets, or evaluation details.
- Only make claims that are supported by the manuscript draft, code, notes, prior paper text, or literature.
- When a detail is uncertain, leave a concise LaTeX TODO comment instead of guessing.
- Be especially careful to distinguish:
  - completed implementation
  - ongoing development
  - proposed future direction

## Literature search policy

Web literature search is allowed and encouraged when it improves the manuscript.

Use web search for:
- papers explicitly mentioned or hinted at in `main.tex`
- URLs or citation clues written in `main.tex`
- foundational or related-work papers relevant to SCIGEN+, DiffCSP++, human preference learning, DPO, diffusion fine-tuning, or band-structure preference evaluation
- filling in missing citation metadata
- expanding related work in a grounded way

When using literature search:
- prioritize authoritative sources such as publisher pages, arXiv, conference proceedings, journals, and official project pages
- add only citations that are genuinely relevant to the manuscript
- do not fabricate BibTeX
- verify title, author list, venue, and year before inserting citations
- keep citation additions targeted rather than bloated
- use URLs in the manuscript only when stylistically appropriate for the existing paper
- prefer integrating literature through the bibliography and citation commands already used in the project

## Editing workflow

For manuscript-editing requests, follow this sequence:

1. Read:
   - `main.tex`
   - `supp_scigenp.tex`

2. Infer:
   - manuscript structure
   - intended contribution framing
   - missing sections or weak transitions
   - where the supplement should support the main text

3. Inspect relevant reference sources as needed:
   - SCIGEN+ code
   - DiffCSP++ code
   - prior SCIGEN paper
   - DPO / diffusion manuscript archives
   - band preference evaluation website and discussion notes

4. Optionally perform literature search:
   - especially for citations referenced or implied in `main.tex`
   - or where related-work coverage is weak

5. Edit only:
   - `main.tex`
   - `supp_scigenp.tex`

6. After editing, report:
   - major narrative and structural changes
   - citations added or updated
   - remaining TODOs
   - any factual uncertainties still needing confirmation

## Style guidance for the agent

- Act like a technical coauthor and scientific editor, not just a copyeditor.
- Prefer meaningful rewrites over superficial sentence polishing.
- Preserve useful macros, labels, and project conventions when possible.
- Avoid unnecessary package changes.
- Do not introduce broken references.
- Keep comments minimal and useful.
- Keep the main paper focused; move overflow detail to the supplement when appropriate.

## Git policy

Unless the user explicitly asks otherwise:

- do not run `git commit`
- do not create commits
- do not push
- do not rewrite git history

You may inspect git diff/status for awareness, but do not make commits unless clearly instructed by the user.

## Default response pattern for manuscript tasks

At the start of the task, briefly state:
- the manuscript structure you inferred
- the main narrative you plan to strengthen
- any high-risk uncertainties

Then perform the edits.

At the end, provide:
- concise summary of file changes
- section-by-section summary of major edits
- TODOs left in the manuscript
- any claims or citations that still need verification
